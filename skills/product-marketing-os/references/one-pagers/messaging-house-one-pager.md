# Messaging House — one page drawn as a house

The reference every copywriter, spokesperson, SDR, and agency briefs from. Pillars become campaign
themes and content pillars downstream — so if a message doesn't hold up the roof or a pillar, it's
cut.

- **When to use:** positioning is signed off and needs to become usable messaging
  (pmm-messaging-positioning / pmm-messaging-hierarchy output); or spokespeople are saying
  different things and need one page to converge on.
- **Length rule:** literally one page drawn as a house — one roof line (8–15 words), exactly 3
  (max 5) pillars, 3–4 proofs per pillar evenly distributed.
- **Lives where:** the wiki (Confluence ships a native template for it); the brief-from page for
  every writer and agency.
- **Hydrating desks:** customer (buyer-pain language, quotes) · product (features, metrics) ·
  competitive (differentiating proof) · analyst-influencer (third-party validation for the
  foundation).

## The six sections are fixed — the ghost generates against them in order

1. Roof: single core message / value prop, 8–15 words max
2. 3 pillars (max 3–5): pillar name + 1–2 sentence benefit + tied to a buyer pain
3. 3–4 proof points under each pillar (metric, customer quote, feature)
4. Foundation: evidence layer — data, case studies, third-party validation
5. Optional walls: voice/tone, dos and don'ts
6. Persona/channel variants noted as pointers, not duplicated on the page

## Proof under every pillar is what turns claims into arguments

- Pillars without evidence are claims, not arguments — every pillar carries its 3–4 proofs.
- Pillars are written in customer pain language, not feature language.
- The consistency test: multiple spokespeople can use the page the same way without coaching.
- It fails when built as a one-off workshop artifact and never activated — it needs quarterly
  review with a protected core.

## The skeleton — one roof, three pillars, evidence in the foundation

```markdown
# [Product] messaging house: [the roof line itself is the governing thought]
*Last reviewed: [date] · Owner: [name] · Consistency-tested with: [spokespeople, date]*

**ROOF — core message (8–15 words):**
> [e.g. "Plotline turns week-long data-queue waits into four-minute answers — no SQL required."]

## PILLAR 1 — [name in buyer-pain language, not feature language]
[1–2 sentence benefit: "[persona] struggles with [pain]; [pillar] means [outcome]."]
- Proof (metric): [number w/ comparator] — Source: [link, date]
- Proof (customer quote): "[verbatim]" — Source: [call/review, date]
- Proof (feature): [capability that delivers the benefit] — Source: [product desk finding]

## PILLAR 2 — [name]
[benefit sentence tied to a buyer pain]
- Proof (metric): [number w/ comparator] — Source: [link, date]
- Proof (customer quote): "[verbatim]" — Source: [call/review, date]
- Proof (feature): [capability] — Source: [product desk finding]

## PILLAR 3 — [name]  *(max 5 pillars; 3 is the norm — cut, don't add)*
[benefit sentence tied to a buyer pain]
- Proof (metric): [number w/ comparator] — Source: [link, date]
- Proof (customer quote): "[verbatim]" — Source: [call/review, date]
- Proof (feature): [capability] — Source: [product desk finding]

**FOUNDATION — evidence layer:** [data set, case studies, third-party/analyst validation]
— Source: [links, dates]

**WALLS (optional):** Voice/tone: [3 adjectives + one banned register] · Do: [...] · Don't: [...]

**Variants:** [persona/channel variants as links — e.g. "→ SDR sequence variant", "→ exec deck
variant" — never duplicated on this page]

**Ask:** [Verb-first — e.g. "Adopt this house for all [Q3] copy by [date] — owner: [name];
quarterly review booked for [date]."]
```
