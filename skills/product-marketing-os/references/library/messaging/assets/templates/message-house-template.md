# Message House Template

**Purpose:** The one-page artifact that circulates — a single umbrella message (the roof) held up by exactly three pillars, each standing on 3-4 proof points, all resting on an evidence foundation. This is the page every copywriter, spokesperson, SDR, and agency briefs from.

**When to use:** After the [messaging canvas](messaging-canvas-template.md) exists. Build the house for a launch, a company narrative, or a product line — one house per narrative. If two messages fight for the roof, build two houses.

**Time to complete:** 1-2 hours of distillation (the thinking already happened in the canvas).

**The format rules (non-negotiable — they are what make it get used):**
- Literally one page, drawn as a house.
- **Roof: ≤15 words.** One core message. Count them.
- **Exactly 3 pillars** (the format tolerates up to 5; it never rewards it). Each pillar: name + 1-2 sentence benefit tied to a buyer pain — pain language, not feature language.
- **3-4 proof points per pillar**, evenly distributed — a metric, a customer quote, or a feature. A pillar without proof is a claim, not an argument.
- **Foundation:** the evidence layer — data, case studies, third-party validation.
- Persona/channel variants are **pointers**, never duplicated on the page.
- **Visible review date.** A house without one reads as stale and gets ignored.
- If a message doesn't hold up the roof or a pillar, it's cut. It can live in the canvas.

## Instructions

1. Copy the roof line from your canvas's primary message. If it's over 15 words, cut words until it isn't — do not negotiate with the limit.
2. Pick the 3 pillars that most directly hold up the roof. Killing a good pillar is normal; parking it in the canvas is where it goes.
3. Write each pillar benefit in the customer's pain language. The test: does it contain a phrase a customer actually said?
4. Attach 3-4 proofs per pillar, each sourced and dated. Distribute evenly — a 6-proof pillar next to a 1-proof pillar means the second pillar isn't ready.
5. Run the **consistency test**: hand the page to two different people; if they deliver the roof and pillars materially differently, the house is ambiguous — tighten it.
6. Date it, name the owner, and book the quarterly review with a protected core (the roof survives reviews; proofs refresh).

## The House

```
                    ┌─────────────────────────────────────┐
   ROOF             │  [Core message — ≤15 words]         │
                    └─────────────────────────────────────┘
                    ┌───────────┬───────────┬─────────────┐
   PILLARS          │ [Pillar 1]│ [Pillar 2]│ [Pillar 3]  │
                    │ [benefit, │ [benefit, │ [benefit,   │
                    │  1-2 sent]│  1-2 sent]│  1-2 sent]  │
                    ├───────────┼───────────┼─────────────┤
   PROOF            │ • [proof] │ • [proof] │ • [proof]   │
   (3-4 each)       │ • [proof] │ • [proof] │ • [proof]   │
                    │ • [proof] │ • [proof] │ • [proof]   │
                    └───────────┴───────────┴─────────────┘
                    ┌─────────────────────────────────────┐
   FOUNDATION       │ [Evidence layer: data, case studies,│
                    │  third-party validation]            │
                    └─────────────────────────────────────┘
```

**Roof:**
> [≤15 words. The one thing.]

**Pillar 1 — [Name]**
- *Benefit (pain-tied):* [1-2 sentences]
- *Proof:* 1. [proof + source] · 2. [proof + source] · 3. [proof + source] · 4. [optional]

**Pillar 2 — [Name]**
- *Benefit:* [1-2 sentences]
- *Proof:* [3-4, sourced]

**Pillar 3 — [Name]**
- *Benefit:* [1-2 sentences]
- *Proof:* [3-4, sourced]

**Foundation:**
[The evidence layer that underwrites the whole house — market data, category stats, case studies, third-party validation]

**Walls (optional):** voice/tone pointer → [voice-and-tone-template.md](voice-and-tone-template.md) · do/don't top 3

**Owner:** [name] · **Reviewed:** [date] · **Next review:** [date] · **Consistency test:** [pass/fail, date]

## Filled example — Plotline

*Distilled from the Plotline messaging canvas; research base dated 2026-06-12.*

**Roof (14 words):**
> **Ask your product a question in plain English — get the answer in four minutes.**

**Pillar 1 — Ask in plain English**
- *Benefit:* The people with the questions can't write the query — so they file a ticket and wait. With Plotline they type the question and get the funnel back, with the generated query shown under it.
- *Proof:*
  1. Neither fetched rival ships plain-English self-serve — Vista leads with the analyst-first suite ("The complete analytics platform for data-driven enterprises"), Pulse with dashboards ("Dashboards your whole team will love"); both route the follow-up "why" through someone who can build the query (homepage fetches, 2026-06-12).
  2. 4-minute median time to first answer — beta telemetry, n=37 accounts (2026-06-12).
  3. 71% of weekly active beta users write no SQL — beta telemetry (2026-06-12).
  4. ⚠️ *Claim discipline: "the only analytics tool that…" does NOT ship until the rival AI-feature sweep (Vista's Copilot beta; Pulse's Insights AI, whose docs page 404'd) is in hand.*

**Pillar 2 — No tickets, no waiting**
- *Benefit:* Buyers describe the status quo as "everything is a ticket" — a ~2-day queue between a question and its answer. Plotline turns that wait into the same afternoon: auto-instrumented events, no analyst in the loop for the common question.
- *Proof:*
  1. "Every 'why did this metric move' becomes a ticket for the data team. By the time it comes back, the sprint's planned." — growth PM, win/loss call (2026-05-28).
  2. ~2-day median queue wait, self-reported across 9 of 14 beta interviews (method: median of stated waits).
  3. No engineering ticket to add an event — the SDK auto-instruments (product demo); vs. Vista's weeks-long, admin-assisted implementations (their onboarding docs + G2, 2026-06-12).

**Pillar 3 — One set of numbers**
- *Benefit:* Every team's dashboard defines "activated" differently, so meetings are spent arguing about whose number is right. Plotline's shared metric catalog defines each metric once — and every answer uses it.
- *Proof:*
  1. "We spent half the QBR arguing about whose activation number was right." — lifecycle marketer, win/loss call (2026-05-28).
  2. One definition per metric, enforced in every Ask answer — the catalog is the source of truth (product demo).
  3. Every answer shows its metric definition and generated query — checkable transparency, the direct counter to "plain-English analytics can't be trusted" (product demo). *(Named gap: the "fewer reconciliation arguments" gain is still unquantified — customer quote owed.)*

**Foundation:**
Why-now: the "be data-driven" mandate reached teams without data departments, and LLM-era UX made plain-English querying an expectation, not a moonshot. Category signal: incumbent suites are bolting AI query onto analyst-first products — those betas are the verification-sweep targets. Beta base: 37 accounts; 4-minute median first answer; 71% of active users write no SQL. Activation proof-in-waiting: % of new accounts that run a real Ask on day one — the single Aha metric (first real answer in under 5 minutes).

**Walls:** voice = transparent-not-magic, on the asker's side, matter-of-fact ([voice-and-tone-template.md](voice-and-tone-template.md)). Don't: AI-magic hype, unscoped "ask anything" accuracy claims, any "only tool" language pre-verification.

**Variants:** personas → [persona-matrix-template.md](persona-matrix-template.md) (PM / lifecycle-PMM / Head of Product).
